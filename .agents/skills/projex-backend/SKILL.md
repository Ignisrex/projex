# projex Backend Implementation

Full task list lives in `plans/backend.md`. Read it first to understand what remains and pick up where work left off.

## Architecture (strictly enforced)

Requests must flow in this exact order — never skip layers:

```
HTTP Handler → Service → Repository → sqlc-generated DB methods (dbgen)
```

- **Handlers** (`server/internal/http/handlers/`): parse HTTP input, call service, write JSON. No business logic, no DB access.
- **Services** (`server/internal/services/`): orchestrate logic, call repositories via interfaces only.
- **Repositories** (`server/internal/repositories/`): all DB access via sqlc methods — no ad hoc SQL.
- **Domain** (`server/internal/domain/`): plain Go structs, no methods or logic.
- **`dbgen`** (`server/internal/dbgen/`): generated by sqlc — never import from handlers or services directly.

All dependencies are wired manually in `server/cmd/api/main.go`. No DI framework.

## Task 1 — Generate sqlc Code

Run this from the repo root before writing any repository code:

```bash
sqlc generate -f ./db/sqlc.yaml
```

This creates `server/internal/dbgen/` with typed Go query methods. Never edit generated files. Re-run whenever `db/queries/` changes.

## Task 2 — Implement ProjectRepository

Create `server/internal/repositories/project_repository.go`:

- Package-private struct `projectRepository` with a `q *dbgen.Queries` field.
- Constructor: `NewProjectRepository(db *sql.DB) ProjectRepository` — initialise `q` once via `dbgen.New(db)`.
- Implement all five `ProjectRepository` interface methods by calling `q.<Method>(ctx, ...)`.
- `domain.Project.ID` is a `string`; the DB uses `bigserial` (`int64`) — parse with `strconv.ParseInt` before passing to sqlc, and `strconv.FormatInt` when mapping back to domain types.
- Return `ErrNotFound` (define in `repositories` package) when `sql.ErrNoRows` is returned.
- Map all `dbgen` row fields to `domain.Project` inside the repository. Upper layers must never import `dbgen`.

## Task 3 — Implement TaskRepository

Create `server/internal/repositories/task_repository.go` using the same constructor pattern:

- Store `q *dbgen.Queries`, initialised once in the constructor.
- Parse all string IDs (`projectID`, task `id`) to `int64` before calling sqlc methods.
- Return `ErrNotFound` on `sql.ErrNoRows`.
- Map `dbgen` rows to `domain.Task`.

## Task 4 — Wire Real Repositories in main.go

Update `server/cmd/api/main.go`:

- Open `*sql.DB` using `DATABASE_URL` from config (`_ "github.com/lib/pq"` or `pgx` driver).
- Call `db.Ping()` at startup — log and `os.Exit(1)` if it fails.
- Replace `NewProjectRepositoryStub()` / `NewTaskRepositoryStub()` with the real constructors.
- Defer `db.Close()`.

## Task 5 — Fix Error Handling in Handlers

Update `server/internal/http/handlers/project_handler.go`:

- Replace all HTTP 501 responses with proper status codes.
- `errors.Is(err, repositories.ErrNotFound)` → 404 with `{"error": "not found"}`.
- All other errors → 500 with `{"error": "internal server error"}`.
- All error responses must use the existing `writeJSON` helper.

## Task 6 — Structured Logging in main.go

`router.go` already has `middleware.Logger` for HTTP request logging — do not change it.

Update `main.go` only:
- Replace `log.Printf` / `log.Fatalf` with `slog.Info` / `slog.Error` (Go's `log/slog`, no third-party package).
- Emit startup events (port, DB connection result) as structured JSON log lines.

## Task 7 — Run Migrations

Before any end-to-end run, apply the schema:

```bash
goose -dir ./db/migrations postgres "<DATABASE_URL>" up
```

## API Conventions

- All responses JSON via `writeJSON(w, status, payload)`.
- Creates → HTTP 201; deletes → HTTP 204 (no body).
- Decode errors → HTTP 400 `{"error": "invalid request payload"}`.
- Route params via `chi.URLParam(r, "id")`.

## Key Config

| Env Var | Default |
|---|---|
| `PORT` | `8080` |
